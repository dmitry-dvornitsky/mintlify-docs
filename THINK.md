## LinkedIn Score — дата анализа

LinkedIn активность меняется со временем — скор устаревает. Нужно показывать когда последний раз запускался анализ для конкретного контакта.

**Варианты:**
1. **Tooltip на LinkedIn Score badge** в таблице Contacts — навёл мышь → `Last analyzed: Jun 12`. Не занимает колонку.
2. **Company/Contact Overlay → Details panel** — рядом с полем LinkedIn: `Score: 72 · analyzed Jun 12`. Предпочтительнее: именно в Overlay SDR решает кому писать в LinkedIn.

**Также:** добавить Quick Filter в Contacts — "High LinkedIn score" / "Has LinkedIn activity". Сейчас SDR вынужден открывать каждый профиль.

## Requests — упрощение

**Текущее:** Import Request = контейнер для группировки нескольких импортов по тематике (например, компании с одной конференции). Внутри реквеста — Companies + Imports табы.

**Проблема:** лишний уровень вложенности. В реальности никто не группирует импорты по тематике — это просто overhead при каждом импорте.

**Идея:** убрать концепцию Import Request, заменить на плоскую **историю импортов** с колонками: дата, тип, количество записей, статус, источник. Контекст источника можно передавать через тег/метку к каждому импорту, не через контейнер.

**Проблема:** Step 2 перегружен — на одном экране одновременно настройки импорта (Import Name, Intent, DNC, Skip), конфигурация обогащения (Revenue filter) и результаты валидации (Valid/Invalid). Непонятная последовательность.

## Campaigns — Template mode: переименование

**Текущее:** "Template mode" — тогл в редакторе последовательности на Step 3.

**Суть фичи:** два принципиально разных режима работы:
- **OFF (по умолчанию):** AI генерирует уникальное письмо для каждого контакта с нуля на основе данных контакта и контекста кампании
- **ON:** SDR пишет структуру сам. Открываются Variables `{}`, Spintax `⇄`, A/B тест. AI только заполняет незаполненные переменные.

**Проблема с названием:** "Template mode" непонятен SDR — не ясно что это режим ручного контроля над письмом.

**Важно:** Message Type отображается и в таблице Campaigns (AI / Template). Переименование нужно применить везде: тогл в редакторе + бейдж в таблице + название режима — один термин по всему продукту.

**Предложения для переименования:**
- **"Write for me / I'll write it"** — переключатель с двумя явными состояниями
- **"Use my template"** — понятно: я сам пишу
- **"Personalize"** (тогл) — ON = AI пишет уникально, OFF = я пишу сам с Variables/Spintax/A/B

Текущая механика: Audiences создаются в Settings, затем выбираются в Find Contacts как First/Secondary Priority. Идея — переиспользовать наборы должностей, не вводить каждый раз вручную.

**Запланировано:** переработать флоу стратегии поиска лидов целиком (Audiences + Find Contacts + Segments). Dmitry хочет изменить концепцию после того как все модули будут задокументированы.

## Campaigns / Settings — Override auto-limits

**Текущее:** Override auto-limits доступен только на уровне отдельного ящика (в сайдбаре Mailboxes → конкретный ящик). Campaign-wide override заявлен как Beta — disabled.

**Проблема:** SDR вынужден идти в каждый ящик отдельно чтобы включить override. Неочевидно что это вообще возможно.

**Идея:** Campaign-wide toggle "Override auto-limits" в Settings кампании с полем числа. Автоматически применяется ко всем ящикам кампании. Ящики с ручным лимитом — показывают предупреждение о конфликте.

---

## Campaigns / Settings — Meetings automation

**Текущее:** "Exclude a company's leads when a meeting is booked" — одна булева настройка без гибкости.

**Проблема:** нет контроля над тем что именно происходит — пауза или полная остановка? На какой срок? Что если нужно продолжить outreach к другим контактам из той же компании?

**Идеи:**
- Выбор действия: Pause / Stop / Do nothing
- Таймаут: пауза на N дней вместо полной остановки
- Scope: только этот контакт / все контакты компании / только контакты без встречи

**Идея:** заменить таблицу на форму с полями прямо в разделе (Name, Description, Competitors). Внизу кнопка **"+ Add another product"** для редких случаев с несколькими продуктами. Продукт как сущность остаётся — просто UX становится проще для типичного случая.
