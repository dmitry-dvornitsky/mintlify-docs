## 2026-07-21

### D-101 · Strategy = Batches + Campaigns, связь через контакты, не через сущности

Финальная сверка модели: стратегия состоит из **Batches** (1..N — история поиска) и **Campaigns** (1..N — потребление лидов). Вопрос — можно ли лидов одного batch использовать в двух разных кампейнах?

**Да.** Эксклюзивность контакта ([D-026](/v4/decisions)) работает на уровне контакта, а не batch: batch — партиция найденных лидов, не резервация под конкретный кампейн. Часть контактов batch уходит в Campaign A, часть — в Campaign B; единственное ограничение — конкретный контакт не может быть одновременно в двух активных кампейнах. При выборе batch как источника в [Step 2 создания кампейна](/v4/workspace/campaigns/create) занятые контакты видны серыми — ровно так же, как при выборе из общего пула стратегии, потому что технически это тот же пул, просто отфильтрованный по batch.

Связь Batch↔Campaign — **many-to-many через контакты**: у стратегии несколько batch и несколько кампейнов, любой кампейн может черпать из любого batch (или из всего пула сразу).

**Почему:** batch решает задачу «что нашли и когда» (история поиска), а не задачу «кому это забронировано» — бронирование уже есть, это эксклюзивность контакта, дублировать её на уровне batch не нужно.
**Где:** [Strategies](/v4/workspace/strategies), [Campaigns · Создание](/v4/workspace/campaigns/create)

### D-100 · Batch — раздел на уровне Campaigns, детальная с воронкой отвала

Вопрос: где хранить batch в навигации — на уровне проекта, рядом со Strategies (отклонено при обсуждении D-099 — batch не самостоятельная сущность проекта, а партия внутри конкретной стратегии) или на уровне Campaigns, внутри детальной стратегии? Второе — да: раз у batch появилась полноценная детальная (воронка + переход в Leads, см. ниже), тихой ссылки под блоком «Каким компаниям» уже недостаточно — нужен раздел **Batches** такого же веса, как раздел [Campaigns](/v4/decisions) (D-098): список batch с diff, контактами, персонами и датой, клик открывает детальную.

Отдельно решили, как из batch попасть к самим лидам и увидеть, где и почему отвалились компании/контакты: детальная batch показывает **воронку конвейера** (Estimate → уже в базе → отправлено в Apollo → прошли колонки квалификации → найдено контактов → прошли DNC → свободны/эксклюзивность → добавлено), с причиной на каждой ступени — реальный отсев (квалификация, DNC) отдельно от простой занятости (эксклюзивность). Компании/контакты самого batch — не отдельная таблица: кнопка «Открыть в Leads →» ведёт в [Leads](/v4/workspace/leads) с фильтром по batch, тот же принцип не-дублирования, что уже решён для [Lists](/v4/decisions) (D-082).

**Почему:** воронка отвечает на «где и почему отвалилось», без которой batch — просто счётчик; фильтр в Leads вместо дубль-таблицы держит одно место правды для просмотра лидов, а не три (Leads/List/Batch).
**Где:** [Strategies](/v4/workspace/strategies), [Strategies · Поиск лидов](/v4/workspace/strategies/find-leads), [Leads](/v4/workspace/leads)

### D-099 · Batch — оценка приблизительная, batch сохраняется как есть

Обсудили проблему: если Batch 2 расширяет критерии Batch 1 (например, company size 11–50 → 11–80), «оценка рынка» каждого batch по отдельности начинает врать — компании, найденные в Batch 1, логически уже входят в более широкую оценку Batch 2, но простое «X из Y» на каждый batch это не показывает.

Рассмотрели три варианта: (A) убрать Batch совсем, оставить плоский лог запусков с выбором диапазона для кампейна; (B) то же + опциональные именованные Presets; (C) ключевать batch по «главным» полям (industry+location), второстепенные — редактировать внутри. Отклонили C: выбор «главных» полей произволен и будет неверным для части стратегий (для одних company size — второстепенная деталь, для других — весь смысл сегмента); блокировка создания нового batch при совпадении ключевых полей заставляет юзера редактировать чужой batch вместо параллельного поиска.

**Решение:** Batch остаётся как сущность — он и так уже решает свою настоящую задачу (группировка запусков с одинаковыми критериями + выбор целого batch как группы контактов для кампейна, [D-096](/v4/decisions); доливка новых лидов в текущий открытый batch тем же Find more). Оценка рынка на запуске/batch — это **приблизительный контекст на момент этого запуска**, не сквозная цифра, которая обязана сходиться между всеми batch стратегии. Не пытаемся математически примирить оценки разных batch друг с другом — Apollo estimate и так не точен по своей природе.

**Почему:** batch решает реальную операционную задачу (группировка + добавление в кампейн + доливка) и её не нужно ломать ради теоретической точности предикта, которая для юзера не так важна, как удобство работы с самими лидами.
**Где:** [Strategies · Поиск лидов](/v4/workspace/strategies/find-leads), [Campaigns · Создание](/v4/workspace/campaigns/create)

### D-097 · Find/Import — авто-квалификация вместо Gate, Import спроектирован, LR override

Разобрали реальный процесс работы с лидами (AS → LR → SDR, «глубокий» и «быстрый» флоу + импорт) и сверили с моделью. Большая часть уже была покрыта (Personas, Find leads, Custom Columns+Enrich, фильтр по колонкам = квалификация [D-074], Lists, Campaign-группа [D-096]) — не хватало трёх вещей:

**1. Gate убран, заменён тихой авто-квалификацией.** У стратегии — опциональные **колонки квалификации** (Custom Columns + значения «подходит»). Заданы → конвейер Find/Import сам прогоняет Enrich по ним и продолжает поиск контактов только по прошедшим компаниям, полностью автоматически, без паузы и без участия человека. Не заданы → конвейер идёт напрямую (это и есть «быстрый» флоу). Разница между «глубоким» и «быстрым» поиском — не ручной шаг в моменте, а то, настроена ли квалификация у стратегии заранее. Отменяет опциональный ручной Gate из [D-025](/v4/decisions).

**2. Import спроектирован** ([Strategies · Import](/v4/workspace/strategies/import-leads), расширяет [D-091](/v4/decisions)): второй вход на стратегии, симметрично Find. **Import companies** (CSV/LinkedIn URLs) идёт в тот же конвейер, что Find — авто-квалификация, поиск контактов. **Import contacts** (готовый список) — сразу в базу с членством в стратегии, без обработки. Флоу загрузки — три шага (Данные → Настройки → Валидация), не один перегруженный экран; история импортов — те же запуски, что у Find, с колонкой Source (Find/Import), без отдельного контейнера-Request (обе идеи взяты из THINK.md и закрыты).

**3. LR может переопределить критерии на конкретный запуск**, в рамках границ, заданных AS, не редактируя сохранённую стратегию. Решает разрыв между «AS создаёт/правит стратегию» (право) и «AS говорит критерии устно, LR операционализирует» (практика) — без передачи LR права редактировать стратегию целиком. Enforcement — доверие/процесс, не техническое ограничение полей.

**Финальная цель («SDR сам жмёт Find leads, LR — только импорт и ручной сложный поиск») уже встроена как дефолт**: стратегия без колонок квалификации — это и есть SDR-дружелюбный автопилотный путь; ничего дополнительно проектировать для этого не нужно.

**Почему:** Gate добавлял ручной шаг ради контроля, который можно получить автоматически, настроив квалификацию один раз при создании стратегии — так проще и по факту эффективнее (не тратим на контакты дисквалифицированных компаний, и никто не должен сидеть у экрана).
**Где:** [Find leads](/v4/workspace/strategies/find-leads), [Strategies · Import](/v4/workspace/strategies/import-leads), [Strategies](/v4/workspace/strategies)

### D-096 · Campaign · Контакты — набор группой; batch-группировка запусков

Ревизия логики Strategies → Leads → Campaign create. Модель уже давала «группу по умолчанию» (пул стратегии, D-024) — не хватало явного описания шага 2 и второго/третьего источника группы. Решено — **три источника, набор группой, не поштучно**:

### D-089 · General — «Контракт» упразднён, условия переезжают в периоды

### D-081 · Do Not Contact — железное правило, автопополнение, Source, уровни блокировки

**Факт (V1):** два таба Companies (домены) / Contacts (email), **+ Add Domain** / **Import CSV**, применяется чекбоксом «Apply DNC» при поиске/импорте (по умолчанию вкл).

Целевое:
- **DNC — железное правило, не чекбокс.** Применяется всегда и приоритетнее любого другого фильтра; чекбокс «Apply DNC» убираем. Выключаемый DNC = риск написать отписавшемуся.
- **Уровни блокировки:** домен компании (вся компания) · **поддомены** домена (eu.acme.com при acme.com) · email контакта. Блокировку «по имени компании» **не делаем** — ложные совпадения.
- **Автопополнение из ответов:** unsubscribe / «не пишите» в Inbox добавляет контакт (и домен по решению) в DNC автоматически с **Added via: Auto** и проставленным Reason.
- **Reason + Added via / by / on** — видно, как и почему запись попала: **Reason** (Unsubscribed/Complaint/Competitor/Existing customer/Client request/Bounced/Other — причина; для Auto ставится сам), **Added via** (Manual/Import/Auto — канал; слово отличается от enrichment-Source в Columns намеренно), **Added by** (кто или System), **Added on** (дата).
- **Удаление с причиной:** Manual/Import удаляются с обязательным reason (право 1–3); **Auto-записи из unsubscribe удалять нельзя** (юридически «никогда не писать» необратимо).
- **Снятие активных на лету:** попадание в DNC останавливает запланированные письма и вынимает контакт из активных кампейнов — как Dropped ([D-066](/v4/decisions)).

**Почему:** «никогда не писать» — обязательство, а не опция; отключаемый галочкой список создаёт юридический и репутационный риск. Автозахват из unsubscribe закрывает главный источник ошибок.
**Где:** [Settings · Do Not Contact](/v4/workspace/settings/dnc)

### D-080 · Навигация: группа = один экран с табами (структура доки ≠ экран 1:1)

Вложенные группы в `docs.json` создаём, когда страницы доки — это **табы/разделы одного экрана-модуля** (Client, Strategies, Campaigns, Columns). Тематически близкие, но самостоятельные экраны Settings идут плоским списком. При этом **структура доки не обязана 1:1 повторять UI**: один экран приложения с табами может дробиться на подстраницы доки, если так удобнее читать (Columns — один экран, но в доке две страницы: Columns + Enrichment History, объединённые группой). Группа именно это и показывает — «это один экран».

**Почему:** группа в сайдбаре сообщает «один экран с вкладками»; дробить объёмный таб на свою страницу доки — нормально, если группа сохраняет связь.

### D-079 · Enrichment History — таб экрана Columns, в доке — отдельная подстраница

Доработка Columns под удобную работу:

### D-050 · Страница описывает только реальный продукт; всё остальное — в TODO

Перезапуск процесса V4 (меняет D-001): страницы перестали отличать существующий функционал от спроектированного — Dmitry путается, что уже есть в продукте. Новый стандарт страницы:

- **«Как работает» и «Правила» описывают только то, что РЕАЛЬНО есть** в продукте. Источник — Dmitry описывает фактическое поведение, Claude уточняет и предлагает улучшения.
- **Раздел «Правила и эдж-кейсы» (таблица «ситуация → поведение») упразднён** — читался тяжело. Вместо него короткие буллеты «Правила» — только фактическое поведение.
- **Чекбоксы внизу страницы = TODO**: только то, что нужно добавить/изменить. Принятые решения — с D-ссылкой; предложения Claude — с пометкой «(предложение)» до решения Dmitry.
- **Страницы компактные, крупные разделы дробятся на подстраницы** — одна подстраница = один фокус.
- Структура сохраняется: Описание → Кто использует → Как работает → Правила → TODO.

Уже написанные V4-страницы перерабатываются по мере прохода: факт остаётся на странице, спроектированное уезжает в TODO со ссылками на решения. Первым полностью закрывается Settings.

**Почему:** спека, смешивающая «есть» и «будет», не отвечает на главный вопрос читателя; таблицы эдж-кейсов не считывались.
**Где:** `V4-REQUIREMENTS.md`, [V4 обзор](/v4), все страницы по мере прохода
