## 2026-07-20

### D-076 · Columns: права, Generate instructions, метка stale вместо версий

Дополнения к [D-075](/v4/decisions): права создания/Enrich — уровни **1–3**; **✦ Generate instructions** (генерация AI-инструкции из Name + Description) — принято, важно. Жизненный цикл значений вместо версионирования колонки: правка инструкции не трогает заполненное; каждое значение помнит `enriched at`, и если инструкцию правили позже — помечается **stale** («по старой инструкции») в ячейке; bulk **Re-enrich stale** освежает только устаревшие. Лёгкая история правок инструкции (кто/когда/что). Архив колонки — по [D-007](/v4/decisions) без спец-логики.

**Почему:** полное версионирование данных — усложнение; метка актуальности значения даёт ту же прозрачность дёшево (паттерн D-018 / D-070 — правка помечает устаревшее, не переписывает молча).
**Где:** [Settings · Columns](/v4/workspace/settings/columns)

### D-074 · Criteria Groups удалены — сливаются в Custom Columns + фильтры

Отдельный модуль критериев квалификации удаляется: это тот же механизм, что **Custom Columns** — поле с AI-инструкцией, заполняемое по сайту/LinkedIn. Критерий «Should» = булева (или числовая) колонка с AI-инструкцией; вердикт «подходит/не подходит» = **фильтр по колонкам** (в 90% случаев критерии нужны именно для фильтрации компаний). Пример «производит жестяные банки И ≥3 локаций» = колонка `Produces tin cans` (bool) + колонка `Locations` (number) + фильтр. Модель Fit/Run/Verdict (обсуждалась в чате) **не строится** — переусложнение; Custom Columns видны в таблице как данные, а не чёрный ящик «да/нет». Именованные наборы условий (бывшие «группы») — опционально Saved Segments, решение отложено («пока пофиг»); отдельная метка «qualified» не нужна.

Отменяет [D-072](/v4/decisions) (переименование) и [D-073](/v4/decisions) (комплексная проработка). AI-квалификация внутри [конвейера поиска лидов](/v4/workspace/strategies/find-leads) — **отдельная механика** ([D-027](/v4/decisions)), не затронута; связь «критерии из ICP → колонки» пересмотреть при проходе Leads.

**Почему:** две сущности с одинаковой механикой (AI-инструкция → значение по сайту) — дубль; фильтр по видимым колонкам понятнее и гибче агрегата-вердикта.
**Где:** [Custom Columns](/v4/workspace/settings/targeting), [Targeting](/v4/workspace/settings/targeting)

### D-073 · Модуль критериев требует комплексной проработки отдельной сессией *(отменено D-074: модуль удалён)*

AI-квалификация компаний недоформулирована — закрывать мелкими решениями нельзя. К проработке отдельным заходом: несколько проверок на одну компанию (разные группы критериев); конфликт вердиктов между группами; переигрывание вердиктов при правке критериев vs перезапуск на затронутых; протухание вердикта во времени; привязка вердикта к паре «компания × критерии» ([D-027](/v4/decisions)). Права раздела — уровни **1–3** (факт).

**Почему:** ядро модуля (жизненный цикл вердикта) не сложилось; частичные решения его зацементируют неправильно.
**Где:** [Settings · Criteria Groups](/v4/workspace/settings/criteria)

### D-070 · Audiences: приоритет — на поиск, не на сущность; версионирование как в KPI

Факты и вердикты: создание/правка — уровни **1–3**; редактировать и архивировать аудиторию можно, **правка не влияет на прошедшие поиски**. Решения: **First/Secondary Priority убираются с сущности Audience** — приоритет проставляется в момент поиска контактов (хранить его на аудитории нелогично: одна аудитория в разных поисках может быть и основной, и запасной). Добавить **Updated at + историю правок** (паттерн KPI). Формализовать **снапшот**: активные стратегии/поиски на старой версии аудитории не ломаются задним числом.

**Почему:** приоритет — свойство конкретного поиска, а не набора должностей; версионирование делает правку безопасной для идущих поисков.
**Где:** [Settings · Audiences](/v4/workspace/settings/audiences), [Поиск лидов](/v4/workspace/strategies/find-leads)

### D-066 · Архивация ящика: Stop + освобождение; перезапуск — вручную из кампейна

Факт-основа: контакт привязывается к ящику **только при первой отправке** — непривязанные контакты архивацией не затронуты и продолжают уходить с живых ящиков обычным распределением. Спец-механики Move нет.

Архивация (после согласия «опасное действие», [D-047](/v4/decisions)) — один шаг со сводкой:

- Контакты с отправленными письмами **без ответа** — стоп, помечаются «прервано: ящик архивирован» и **освобождаются** от эксклюзивности.
- **Ответившие не перезапускаются никогда** — треды остаются read-only.
- **Перезапуск — явное действие в кампейне**: алерт «N контактов прерваны из-за архива ящика» → **✦ Вернуть в набор** (сиквенс с нуля, новый тред, живой ящик; AI-кампейн пишет новое первое письмо — не дубль; Static — предупреждение «получат тот же темплейт повторно») или **Отпустить** (остаются свободными лидами, алерт закрывается).
- Уведомление об архивации с числом прерванных — получателям из [D-065](/v4/decisions).
- Дополнение (в тот же день): прерванные контакты **явно видны в кампейне** — статус **Dropped** в списке контактов кампейна + фильтр по нему, не только алерт.

**Почему:** архивирует ящик EDS по инфраструктурной причине, а «написать заново» — решение владельца кампейна в его контексте; между событиями часто нужна пауза; переиспользуются существующие механики (освобождение + досыпание в набор) вместо новой.
**Где:** [Settings · Mailboxes](/v4/workspace/settings/mailboxes), [Campaigns](/v4/workspace/campaigns)

Оверлей сотрудника: Info · Projects · Activity (лента из audit log). **Deactivate — мягкая**: доступ снят, юзер вне выборок active users, история и ссылки остаются; Reactivate возможен.

### D-054 · Состояние проекта: Status · Phase · Health вместо двух смешанных списков

Три поля, каждое отвечает на один вопрос:

- **Status** («шлём ли письма», обязательный, шапка проекта, правка 1–2): **Pending · Active · On Hold · Closed**. К On Hold/Closed обязательна причина: Pending — Delayed Start; On Hold — Delayed Payment / Client Request; Closed — Completed / Churn. On Hold и Closed останавливают отправку.
- **Phase** («где клиент на пути», движется вперёд, руками 1–2, в General): **Onboarding · Adoption · Ongoing · Working Off**.
- **Health** («как себя чувствует», скачет, с комментарием): **Good · Normal · At Risk**. Позже — кандидат на автоматический расчёт (reply rate, KPI pace, просрочки).

Маппинг старых значений: In Progress→Active; Paused/On Hold/Delayed Payment→On Hold+причина; Done→Closed·Completed; Churn (был в обоих списках)→Closed·Churn; Delayed Start→Pending+причина; Active Ongoing→Phase Ongoing; Active Loyal→Health Good; Active Struggling/Predicted Churn/Predicted Working Off→Health At Risk с пометкой.

Соответствие с Draft/Archived из спеки жизненного цикла — уточнить при проходе Admin → Projects.

**Почему:** 18 значений в двух списках смешивали три измерения (машина / этап пути / оценка) — Churn существовал в обоих; комбинации трёх коротких полей выражают то же точнее.
**Где:** [Settings · General](/v4/workspace/settings/general), [жизненный цикл](/v4/admin/projects/lifecycle), [Флоу](/v4/flows)

### D-029 · Два типа кампейна: AI Generated и Static, строго

Тип выбирается при создании и не меняется; миксовать шаги двух типов в одном кампейне нельзя.

**Почему:** строгий тип — меньше состояний и проще объяснять; Static закрывает клиентов с жёстким контролем текста, спинтакс даёт им вариативность для deliverability.
**Где:** [Campaigns](/v4/workspace/campaigns), [Создание](/v4/workspace/campaigns/create)

### D-009 · Wireframe UI Kit: интерфейс работает без текста

Все макеты V4 строятся на минималистичном wireframe-ките с тройным кодированием смысла: **форма + иконка + цвет**, текст только дублирует. База — grayscale; пять семантических цветов (синий — действие, зелёный — готово, янтарный — ожидание, красный — опасное, фиолетовый — AI), один цвет = один смысл. Контрольный тест — «проверка китайским»: подписи на незнакомом языке не должны мешать пользоваться экраном.

**Почему:** язык-независимый интерфейс проверяет чистоту UX — если элемент понятен только по подписи, он спроектирован плохо; заодно решает доступность (дальтонизм) и снижает нагрузку на локализацию.
**Где:** [UI Kit](/v4/ui-kit) — все элементы и чеклист применения для каждого нового экрана.

### D-008 · Ящик: один проект, много кампейнов

Email-ящик принадлежит ровно одному проекту, но обслуживает любое число его кампейнов. Дневной лимит считается глобально по ящику; кампейны делят его между собой.

**Где:** [Settings · Mailboxes](/v4/workspace/settings/mailboxes)

### D-007 · Hard delete запрещён во всём приложении

Только soft delete: сущности архивируются/завершаются и скрываются из рабочих списков, данные не теряются никогда. Стратегии, кампейны, лиды, инвойсы, проекты — не удаляются. Кнопка «Delete» в продукте означает архив.

**Исключения (единственные два, уточнены 2026-07-15):**
1. **GDPR-запрос** — строгий админ-процесс в [Admin → System](/v4/admin/system): физически стираются **персональные данные** (имена, email, телефоны, профили) клиента или лидов, но их **статистика сохраняется анонимизированной** — метрики кампейнов, воронки и отчёты не ломаются.
2. **Draft-сущности любого типа** — не только проект: черновик кампейна, письма, стратегии и т.д. можно удалить по-настоящему, пока он не запущен/не отправлен. Draft без данных — мусор, а не история.

**Почему:** данные — история работы с клиентом; восстановить удалённое дороже, чем хранить архив.
**Где:** применяется на каждой странице V4 — базовый чеклист в [обзоре](/v4) обновлён.

### D-006 · Workspace работает одинаково для всех ролей

Home строго личный: свои проекты + своя очередь. Нет проектов — Home пуст, без спецблоков для менеджмента. Хочешь открыть проект, в котором ты не в команде, — иди в [Admin → Projects](/v4/admin/projects), проект откроется в read-only с баннером.

**Почему:** один и тот же экран для всех ролей проще; портфельный обзор — задача Admin, дублировать его в Home нельзя.
**Следствие:** эскалации до менеджмента идут через Command Center и Slack-уведомления, не через Home.
**Где:** [Home](/v4/workspace/home) · [Layout](/v4/layout) · [Admin → Projects список](/v4/admin/projects/list)

### D-005 · Логин всегда ведёт в Home

Даже если у пользователя один проект — никакого автовхода в Dashboard.

**Почему:** очередь дел должна жить на одном и том же месте независимо от числа проектов.
**Где:** [Layout](/v4/layout) · [Home](/v4/workspace/home)

### D-004 · Ask AI — постоянная кнопка в хедере

Правая панель с контекстом текущего экрана: отвечает на вопросы по данным, выполняет действия строго в правах пользователя через Gates.

**Где:** [Layout](/v4/layout) · [AI & Automation](/v4/platform/automation)

### D-003 · Мобильной версии нет

Продукт — desktop web, минимальная ширина ~1280px, на узких экранах — предупреждение. Срочные события доходят через Slack-уведомления.

**Где:** [Layout](/v4/layout)

### D-002 · Projects убран из навигации Workspace

Delivery попадает в проекты через Home и project switcher. Управление портфелем (список, создание, жизненный цикл) — в Admin → Projects. Дашборд проекта остаётся в Workspace как первый пункт проектного сайдбара.

**Почему:** список проектов нужен только управляющим ролям; для delivery это лишний раздел.
**Где:** [Admin → Projects](/v4/admin/projects) · [Dashboard](/v4/workspace/dashboard) · [Layout](/v4/layout)

### D-001 · Стандарт V4: единый формат страницы с чеклистами

Каждая страница: вступление → Кто использует → Как работает → Правила и эдж-кейсы → Чеклист функционала с чекбоксами. Страницы маленькие, большие модули дробятся на подстраницы. Полный жизненный цикл каждой сущности обязателен (создание, правка, удаление, архив, bulk). Claude обязан проактивно предлагать эдж-кейсы.

**Где:** [V4 обзор](/v4) · `V4-REQUIREMENTS.md` в корне репозитория
