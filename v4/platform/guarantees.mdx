Всё перечисленное здесь действует **в каждом модуле V4 по умолчанию** и не повторяется в чеклистах страниц ([D-033](/v4/decisions)). Чеклист страницы модуля содержит только специфичное для модуля; если модуль **отклоняется** от гарантии или уточняет её (например, «удаление из DNC — только AS, с причиной»), это фиксируется в его «Правилах и эдж-кейсах».

## Жизненный цикл данных

- **Hard delete запрещён** ([D-007](/v4/decisions)): «удаление» = архив; данные и статистика сохраняются, разархивирование возможно. Исключения — только GDPR-процесс ([Admin → System](/v4/admin/system)) и Draft-сущности.
- Архивирование спрашивает подтверждение и объясняет судьбу связанных данных.
- Где есть версии — история хранится целиком, откат создаёт новую версию (история линейна).

## Права

- Права проверяются по **уровню доступа** (в продукте — поле **Role**, текстом): **1** Admin · **2** Manager (AS, PM, Sales) · **3** Member (SDR, LR, EDS, Content Writer) · **4** Client ([D-040](/v4/decisions), нумерация [D-051](/v4/decisions), состав [D-061](/v4/decisions)/[D-062](/v4/decisions)); в текстах спеки права пишутся цифрами («правка — 1–2»). Позиция человека — поле **Title**, описывает workflow, не права. Единственный источник — [матрица](/v4/platform/roles).
- Недоступные **разделы скрываются**, недоступные **действия — disabled с тултипом** «у кого есть право» ([D-017](/v4/decisions)).

## Audit log

- Каждое значимое изменение (создание, правка, смена статуса, архив, действия с деньгами и правами) пишется в audit log: кто, что, когда. Единый лог с фильтрами — [Admin → System](/v4/admin/system).
- Действия AI-агентов логируются наравне с людьми, с пометкой агента.

## Состояния интерфейса

- **Empty state** у каждого списка: объяснение, зачем объект нужен, + CTA первого действия.
- **Loading / error** — скелетоны и поблочные ошибки, не белый экран целиком.
- Необратимое действие — подтверждение; обратимое — toast с **Undo** (10 сек, [Layout](/v4/layout)).

## Работа со списками

- Поиск, фильтры, сортировка по колонкам; пагинация или ленивая подгрузка.
- Настройка колонок и CSV-экспорт — у полноценных таблиц (не у вспомогательных списков).

## Валидация и лимиты

- Валидация на вводе с внятной ошибкой **до** старта операции (не после).
- Лимиты (количество, длина, размер файла, квоты тарифа) показываются заранее; конфликт лимитов — в момент ввода.
- Имена объектов уникальны в рамках проекта.

## Уведомления

- Описываются **только** в [карте уведомлений](/v4/platform/notifications): событие → кому → канал. Страницы модулей на карту ссылаются, но собственных списков уведомлений не ведут.

## Конкурентная работа

- Правки сохраняются автоматически; при одновременной правке побеждает последняя запись, история версий страхует ([D-021](/v4/decisions)).
- Потеря доступа/соединения на лету не теряет введённое ([Layout](/v4/layout)).

## Тултипы и микрокопия

- У каждого icon-only контрола есть тултип; у текстовых кнопок — только при неочевидных последствиях ([D-009](/v4/ui-kit)).
- Конкретные тексты тултипов и кнопок — задача дизайна и кода, **не спеки** ([D-034](/v4/decisions)): спека описывает поведение, не строки.

## Чеклист платформенных гарантий

Реализуется один раз на уровне платформы; страницы модулей эти пункты не дублируют.

- [ ] Архив вместо удаления везде + разархивирование (D-007); GDPR-процесс и Draft-исключение
- [ ] Скрытие разделов / disabled-действия с тултипом по матрице ролей (D-017)
- [ ] Audit log значимых изменений, единый просмотр в Admin → System
- [ ] Empty / loading / error states по стандарту
- [ ] Undo-toast для обратимых, подтверждение для необратимых
- [ ] Поиск, фильтры, сортировка, пагинация у списков; колонки и CSV у таблиц
- [ ] Валидация на вводе, лимиты заранее, уникальность имён
- [ ] Уведомления только через карту
- [ ] Автосохранение, last-write-wins + версии
- [ ] Тултипы по правилу D-009
