# V4 — Требования и план (НЕ ТЕРЯТЬ)

Финальная версия спецификации продукта. Пишется с нуля, но вбирает **все** наработки:
V1 (реальный функционал), V2 (Admin, роли, автоматизация, Client Portal), V3 (guided-флоу, простота), THINK.md (идеи), redesigns/, roadmap/. V2/V3/redesigns заморожены и убраны из сборки, V1 помечается deprecated помодульно ([D-035](v4/decisions.mdx)); открытые вопросы живут на страницах модулей ([D-036](v4/decisions.mdx)).

## Зафиксированные требования

1. **Один стандартный формат страницы** — каждая страница V4 строится по единому шаблону (см. ниже). Никаких страниц «в свободной форме».
2. **Чеклист с чекбоксами внизу каждой страницы** — список функционала фичи в виде `- [ ]`. Это одновременно: спека для разработки, чеклист приёмки и способ увидеть, что забыто. В чеклист попадает **только специфичное для модуля** — сквозные гарантии зафиксированы один раз в `v4/platform/guarantees.mdx` (D-033) и не повторяются.
3. **Страницы маленькие** — мало текста, одна фича = одна страница. Большой модуль дробится на подстраницы (обзор модуля + страница на каждую фичу).
4. **Полное покрытие жизненного цикла каждой сущности** — создание, просмотр, редактирование, архив, дубликат, bulk-операции. Ни одна сущность не описывается без ответа «а как её убрать и что при этом происходит со связанными данными».
4a. **Hard delete запрещён во всём приложении (D-007)** — только soft delete/архив; исключения — GDPR-процесс и Draft-сущности. Детали — в журнале `v4/decisions.mdx`.
5. **Два контура обязательны**: Delivery (рабочее пространство команды) и Admin (менеджмент и C-level). Плюс Client Portal как третий, клиентский контур.
6. **Процесс работы**: идём модуль за модулем. На каждом модуле Claude ОБЯЗАН проактивно предлагать упущенные мелочи и эдж-кейсы — это часть задачи, а не опция.
7. Терминология и решения из THINK.md применяются сразу (например: Requests → плоская история импортов; Template mode → переименовать; Audiences → в контекст поиска).
8. **Реестр базовых флоу** — `v4/flows.mdx` (D-045): для каждой фичи перечислены пользовательские флоу «что можно сделать с сущностью». Модуль не считается описанным без флоу в реестре; новые флоу вбрасываются сначала туда (Dmitry диктует — Claude раскладывает), дыры помечаются ⚠️.

## Стандартный шаблон страницы (D-050 — «факт vs TODO»)

```mdx
---
title: "<Название фичи>"
description: "<одно предложение — что делает>"
---

<1–2 предложения: что это и зачем>

## Кто использует
<роль — что делает; компактно>

## Как работает
<ТОЛЬКО то, что реально есть в продукте; компактно, скрин где есть>

## Правила
<короткие буллеты фактического поведения: лимиты, права, «что происходит при X».
Таблиц «ситуация → поведение» больше нет — не считывались>

## TODO
- [ ] <только то, что нужно ДОБАВИТЬ/изменить; решения — с D-ссылкой,
      идеи Claude — с пометкой «(предложение)» до решения Dmitry>
```

Правила шаблона:
- **Страница = реальный продукт.** Всё несуществующее живёт только в TODO. Смешивать нельзя — это главная причина перезапуска (D-050).
- Источник фактов — Dmitry описывает, как работает сейчас; Claude уточняет, предлагает улучшения и упущенное — в TODO.
- Компактность: легко считывается с одного взгляда; крупный раздел дробится на подстраницы (одна подстраница = один фокус).
- Ссылки между страницами обязательны (сущность упомянута → ссылка на её страницу).
- Тексты тултипов и кнопок в спеке не фиксируются (D-034). Сквозные гарантии — один раз в `v4/platform/guarantees.mdx` (D-033), в TODO не дублируются.

## Платформенные гарантии (вместо базового чеклиста на каждой странице)

Сквозные требования — жизненный цикл сущности (архив вместо удаления, дубликат, bulk), состояния интерфейса, работа со списками, права по матрице, audit log, конкурентное редактирование, валидация и лимиты, уведомления через карту — зафиксированы **один раз** в `v4/platform/guarantees.mdx` (D-033) и действуют на каждой странице по умолчанию. На странице модуля описываются только специфичные ответы («что происходит со связанными данными при архиве именно этой сущности») и отклонения от гарантий — в «Правилах и эдж-кейсах».

## Сквозные эдж-кейсы (проверять в каждом модуле)

Глобальный список — то, что чаще всего забывается:

| Тема | Вопросы, на которые каждая страница должна отвечать |
|---|---|
| **Удаление с зависимостями** | Удалили стратегию — что с её кампаниями? Удалили ящик — что с активными кампаниями на нём? Удалили сотрудника — кому переходят его лиды/задачи/треды? |
| **Редактирование запущенного** | Можно ли править сиквенс работающей кампании? Что видят контакты, уже прошедшие шаг? |
| **Дубликаты данных** | Один лид найден дважды / импортирован повторно / есть в двух кампаниях — merge, приоритет, блокировка |
| **Отключение каналов** | Ящик отвалился (пароль, бан), LinkedIn-аккаунт заблокирован, календарь отозван — что с кампаниями, кто получает алерт |
| **DNC и unsubscribe** | Приоритет DNC над всем; unsubscribe задним числом; DNC на уровне компании vs контакта; GDPR-удаление по запросу |
| **Встречи** | Reschedule, no-show, cancel лидом, cancel нами, перенос таймзоны, двойное бронирование слота |
| **Деньги** | Просроченный инвойс — что блокируется? Refund, смена тарифа посреди месяца, пауза подписки, конец пилота |
| **Команда** | Onboarding/offboarding сотрудника, передача клиента между SDR, отпуск (кто подхватывает Inbox) |
| **Время** | Таймзоны клиента/лида/ящика, рабочие часы отправки, праздники |
| **AI-действия** | Где AI действует сам vs ждёт аппрува; как откатить действие агента; лог действий AI отдельно от людей |
| **Клиент в портале** | Что клиент видит / не видит; клиент отклонил аппрув — что дальше; клиент ушёл — retention данных |
| **Лимиты роста** | 3+ продукта у клиента, 50+ кампаний, 100k лидов — где UI ломается |

3. Порядок модулей (обновлён 2026-07-15): ~~Projects~~ ✅ → ~~Layout~~ ✅ → **Settings → Client → Strategies → Leads → Campaigns → Inbox → Meetings → Tasks → Overlay → Home (почти последним — агрегирует всё) → Admin-модули → Portal → Платформа** (Notifications последним — карта собирается из чеклистов модулей).

6. **Каждое решение Dmitry записывается в `v4/decisions.mdx`** (журнал D-001, D-002, …): дата, суть, почему, ссылки на страницы. Решения не удаляются — пересмотренные помечаются «заменено D-XXX». Журнал — единственное место: ни в этом файле, ни на страницах решения не пересказываются (D-036).
